## A row delete is not an erasure

In a plain database, “delete the user” is one statement. In a RAG system, a user's data is scattered across representations:

- the **facts** about them,
- the **embeddings** of those facts (and of any chunk that mentions them),
- the **graph edges** connecting them to other entities,
- **derived artifacts** — summaries, profiles, digests built earlier.

Delete the source row and the user is still findable by vector similarity, still reachable by walking an edge, still named in a cached summary. The regulator's question is “is the data gone?” — and “the main row is gone” is not a yes.

## What a real forget removes

Erasure has to cascade through every representation tied to the entity:

1. **Facts** — every claim where the entity is subject or object.
2. **Edges** — every typed relationship in or out.
3. **Embeddings** — the vectors, not just the text.
4. **Derived artifacts** — anything compiled from the above.

After the cascade, a similarity search for the user returns nothing, and a graph walk finds no node. That is the bar.

## Keep the proof, not the data

You still have to *prove* you erased the user — without keeping what you erased. The pattern is a **tombstone**: an HMAC of the entity id plus a timestamp. It carries no recoverable personal data, but it lets you answer an audit and it blocks a later re-ingest from quietly recreating the entity under the same id.

So the invariant after forget is precise: **data gone, proof kept.**

## Forget is not retract

These get confused, and the confusion is a compliance bug.

- **Retract** — the fact was wrong. Mark it retracted, keep it in the audit trail, exclude it from live answers. History is preserved on purpose.
- **Forget** — the person has the right to be gone. Destroy every representation. History ends.

Use retract for corrections, forget for erasure. Never the reverse.

## How Brain implements it

In INITE Brain, `POST /v1/entities/:id/forget` runs a synchronous hard cascade — facts, edges, and embeddings deleted in one operation — and writes an HMAC tombstone. The memory-lifecycle eval gate asserts the invariant and refuses to pass below 1.0: after a forget, the entity is unretrievable by any path, and the tombstone is the only thing that remains.
